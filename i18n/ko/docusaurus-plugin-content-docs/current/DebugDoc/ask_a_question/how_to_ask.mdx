# 질문하기 전에 먼저 읽어주세요

  * QQ/위챗 등의 커뮤니케이션 도구는 빠르고 편리한 대화 수단을 제공하지만, 문제가 발생했을 때 아래 단계를 따르는 것이 좋습니다.
  * 궁극적으로 유용한 답변을 얻을 수 있는지 여부는 당신이 질문하고 추가 질문하는 방식에 따라 달라집니다.

## 문제 해결 단계:

  1. 스스로 해결 시도: 자신의 지식을 활용하여 문제를 먼저 해결해 보세요. 영어 출력 정보에 겁먹지 말고 최대한 이해해 보세요. 대부분의 정보 출력은 방향성을 가지고 있으며, 그것을 찾고 스스로 해결하려고 노력하면 됩니다.
  2. 온라인 검색: 온라인에서 유사한 문제를 겪은 사람이 있는지 검색해 보세요. 만약 있다면, 먼저 다른 사람의 해결 방안을 참고하고, 없다면 그룹에서 질문하세요.
  3. 그룹에서 질문: 그룹에서 질문할 때는 먼저 문제를 명확하게 설명하고, 스크린샷을 첨부하여 모두가 문제를 빠르게 파악할 수 있도록 하세요. 제공하는 정보가 많을수록 문제 해결의 효율성이 높아집니다.

## 질문 시, 다음 사항을 최대한 지키면 문제가 빠르게 해결될 확률이 크게 높아집니다:

### 항상 답변자의 관점에서 문제를 생각하세요

  매번 질문할 때마다 답변자의 입장에서 생각해 보세요:
  * 답변자들은 모두 대가이므로 그들의 시간은 귀중합니다.
  * 어떻게 하면 모두가 한눈에 내 질문을 이해할 수 있게 할 수 있을까요? 최대한 기본적인 질문에 답변자가 질문하지 않도록 해야 합니다.
  * 어떻게 하면 모두가 내 질문에 답하고 싶어할까요? 예를 들어, 기술적 토론의 깊이가 있거나, 답변자에게 존경받는 느낌을 주거나, 직접 돈을 줄 수 있는지 등.
  * 답변자가 내가 문서를 읽지 않았거나, 코드를 보지 않았거나, 로그를 확인하지 않았거나, 검색하지 않았거나, 생각하지 않았거나, 시도하지 않았거나, 어떤 노력도 하지 않고 바로 질문했다고 느끼게 하지 마세요.

### 문제를 정리하고, 무엇이 일어났는지, 내가 무엇을 했는지 명확히 하세요:

  * 어떤 효과나 기능을 구현하고 싶나요?
  * 이 효과를 구현하기 위해 어떻게 했나요? 구체적인 과정은 무엇인가요?
  * 실행 과정에서 어떤 오류가 발생했나요? 어떤 현상이 나타났나요(예: 오류가 발생했는지, 어떤 오류가 발생했는지, **완전한** 오류 내용은 무엇인가요? 마지막 몇 줄만 보지 말고 최대한 앞의 로그를 확인하세요)
  * 오류 정보를 진지하게 읽었나요? 오류 정보에 오류의 원인과 해결 방법이 힌트로 제공되었나요?
  * 이러한 오류 정보를 바탕으로 문제를 해결할 수 있는지 신중히 고려해 보세요.
  * 문서, FAQ, 이슈, 검색 엔진을 통해 문제 해결 방법을 검색할 수 있는지 확인하세요.
  * `문서가 어디 부족하다고 느껴지면 직접 지적해 주세요!! 다른 걱정하지 말고, 제가 대가 아니니 생각이 미치지 못한 부분이 있을 수 있습니다!!`

  * 질문: 대가님, 이 문제 어떻게 해결하죠?

### 문제점

  * *무작위 화면 캡처* 화면 캡처 기능을 사용할 수 있다면 스크린샷을 사용하세요. 특별한 경우가 아니라면 화면을 좀 더 명확하게 찍어주세요.
  * *정보 부족* CAN ID를 찾지 못하는 문제는 여러 가지가 있을 수 있습니다. 예를 들어, 선을 잘못 연결했거나, 상위 기기와 하위 기기의 속도가 다르거나 등. 구체적인 상황에 문제를 한정하여, 구체적인 조건을 제공하면 문제를 좁히는 데 도움이 됩니다.
  * 스크린샷을 찍는 방법을 모른다면 이 [스크린샷 튜토리얼](https://www.bilibili.com/video/BV1LB4y1t7jy/?spm_id_from=333.337.search-card.all.click&vd_source=36affb37d7228751cd1fcbbda487dfdf)을 참조하세요.
## 질문 템플릿

질문을 최대한 우아하게, 필요 없는 감정 표현이나 불평의 단어를 추가하지 말고, 매 단어와 구두점을 신중하게 선택하여, 답변자의 입장에서 문제를 생각하고, 답변자가 빠르게 도울 수 있도록 하세요. 글자가 너무 적으면 설명이 부족하고, 너무 많으면 인내심을 잃게 할 수 있습니다.

### 문제를 스스로 해결할 수 없을 때, 고려해야 할 사항:

  * 누구에게 질문할지, 어디서 질문할지, 누가 내 질문에 더 답할 가능성이 높은지, 그리고 실시간으로 답할 수 있는지?
  
  * 그에게 어떤 정보와 현상을 제공해야 빠르게 문제를 해결할 수 있게 할 수 있을까요?
    * 내 목적을 제공합니다. (답변자가 내가 무엇을 하고 있는지 알 수 있게 합니다)
    * 완전한 실행 과정과 과정에서 나타난 현상을 제공합니다. (답변자가 내 과정을 따라 하여 문제를 재현할 수 있게 합니다)
    * 오류가 발생한 부분을 지적하고, 어디서 예상과 다른 결과나 현상이 나타났는지 명확히 합니다! (답변자가 예상과 다른 부분을 알 수 있게 합니다)
    * 오류 정보를 제공해야 합니다. 가능한 한 더 많은 스크린샷, 더 많은 로그를 제공하세요. 작은 스크린샷 하나만 찍거나 로그의 일부만 제공하지 마세요. (답변자가 오랫동안 이 문제를 다루지 않았을 수 있기 때문에, 스크린샷과 완전한 로그로 빠르게 기억을 되살릴 수 있게 하며, 세부 사항을 통해 문제가 어디서 발생했는지 빠르게 파악할 수 있게 합니다)

  * 어떻게 하면 질문이 진지하게 보이고, 설령 내가 초보자여도 모두가 기꺼이 답해주고 싶어할까요?

### 제목

  어디서든 질문할 때 자신의 문제에 대해 30자 내외의 제목을 만들어, 문제의 핵심 아이디어를 명확히 하세요:
  * 문제의 종류, 문제 질문, 버그 제출, 경험 공유 등을 포함합니다. 모두가 텍스트로 가득 찬 화면에서 즉시 내가 무엇을 원하는지 알 수 있게 합니다.
  * 문제의 핵심을 한마디로 설명합니다. 예를 들어, **CAN ID를 찾지 못하거나, 메인보드 연결이 되지 않는 등**

  그래서 종합적으로 제목은 이렇게 될 수 있습니다:
  * `【질문】 SHT36V3이 CAN ID를 찾지 못했는데, 이미 확인한 사항은 무엇 무엇입니다`

  이런 제목은 **절대** 다음과 같이 나타나면 안됩니다:
  * `아아아아아 왜 내 보드가 ID를 못 찾죠` (모르죠, 당신이 어떻게 연결하고 어떻게 했는지 알 수 없으니)
  * `왜 내 상위 기기가 시작되지 않죠` (모르죠, 어떤 상위 기기인지 알 수 없으니)
  * `왜 장치가 없죠` (모르죠, 어떤 제품인지 모르니)
  * `이 오류는 무슨 오류죠` (문서를 안 읽고 와서 바로 질문)
  * `메인보드 받았는데, 드라이버 어떻게 연결하죠?` (문서를 안 읽고 와서 바로 질문)
  * `이 선 어떻게 연결하죠` (무슨 문제인지)
  * `DFU에 들어가고 나서 왜 못 불러들이죠, DFU에 들어갈 수가 없어요 등등` (당신이 어떻게 했는지, 어떻게 연결했는지 알려줘야 합니다. 아니면 다른 사람들이 당신이 헛소리한다고 생각할 수 있습니다)

  이렇게 질문할 수 있습니다:
  * `【질문】 내 보드가 전원을 반대로 연결한 후 지금은 보드가 시작되지 않습니다. 보드가 어디서 타버렸는지 어떻게 판단할 수 있나요? 가능하다면 어떻게 복구할 수 있을까요?`
  * `【버그】 이 문서 여기 설명이 부족합니다. 비슷한 내용을 추가하거나 수정해 주셨으면 합니다.`

### 참고 템플릿

  * 질문을 정리하는 데 시간을 조금 들여 다음 정보를 명확히 설명해 주세요:
  * 궁극적으로 유용한 답변을 얻을 수 있는지 여부는 당신이 질문하고 추가 질문하는 방식에 따라 달라집니다.

  1. **무슨 문제**     예를 들어: 새로 산 메인보드와 도구판에서 장치를 찾지 못함
  2. **어떤 하드웨어 장치**  예를 들어: 셋톱박스+D5 메인보드+SHT36V3
  3. **어떻게 연결**     예를 들어: 연결 사진을 고화질로 대가님에게 보내기
  4. **어떤 문제 현상**  예를 들어: 상위 기기에서 **lsusb**를 입력해도 장치가 없음 등. 완전한 정보 스크린샷을 첨부하세요.

  【질문】 새로 산 D5 메인보드와 SHT36을 셋톱박스 상위 기기에서 **lsusb**를 사용해도 장치를 찾지 못합니다.
